# Reverb Multiapplication Manager - Copilot Instructions

## Project Overview

This is a **Laravel Reverb Multiapplication Manager** - a centralized dashboard for managing multiple Laravel applications and monitoring their real-time performance using Laravel Pulse. The system manages WebSocket connections, application health, and provides performance insights across all connected applications.

## Tech Stack

### Backend
- **Laravel 12** (PHP 8.2+)
- **Laravel Reverb** - WebSocket server for real-time communication
- **Laravel Pulse** - Application performance monitoring
- **Laravel Fortify** - Authentication with 2FA support
- **SQLite/MySQL** - Database (configurable)
- **Pest** - Testing framework

### Frontend
- **Vue 3** with TypeScript
- **Inertia.js** - Modern monolith SPA architecture
- **Tailwind CSS 4** - Utility-first styling
- **Reka UI** - Accessible component library (Radix Vue port)
- **Lucide Icons** - Icon system
- **VueUse** - Composition utilities
- **class-variance-authority (CVA)** - Component variant management

### Build Tools
- **Vite** - Frontend bundler
- **Laravel Pint** - PHP code style fixer
- **ESLint + Prettier** - JavaScript/TypeScript linting and formatting

## Architecture Principles

### 1. **Real-Time First**
- Use Laravel Reverb for all real-time features (app status, metrics, logs)
- Implement broadcasting events for state changes
- Keep WebSocket connections efficient and stable

### 2. **Dynamic Application Provider**
- Applications are stored in the database, not config files
- `DatabaseApplicationProvider` reads from the `applications` table
- Reverb authenticates clients using database credentials
- 5-minute cache with automatic invalidation on changes
- See [REVERB_INTEGRATION.md](REVERB_INTEGRATION.md) for full details

### 3. **Performance Monitoring**
- Integrate Laravel Pulse for performance metrics
- Track key metrics: response time, throughput, memory usage, queue jobs
- Display aggregated metrics across all managed applications
- Implement custom Pulse recorders for application-specific metrics

### 4. **Multi-Tenancy Pattern**
- Each managed application is a separate entity with its own configuration
- Store application credentials securely (encrypted in database)
- Support multiple database connections
- Isolate application data and metrics
- Reverb authenticates each app using `DatabaseApplicationProvider`

### 5. **Type Safety**
- Use TypeScript for all frontend code
- Define proper interfaces for all data structures
- Leverage Laravel's type hints in PHP
- Use shared types between backend and frontend (Inertia props)

## Code Style & Standards

### PHP
- Follow PSR-12 coding standards (enforced by Pint)
- Use typed properties and return types everywhere
- Prefer Laravel's built-in features over third-party packages
- Use actions/services for complex business logic
- Keep controllers thin - delegate to actions/services
- Write feature tests with Pest for all critical paths

### TypeScript/Vue
- Use Composition API with `<script setup>` syntax
- Leverage TypeScript strict mode
- Use composables for shared logic (place in `resources/js/composables/`)
- Follow Vue 3 best practices (defineProps, defineEmits)
- Use Reka UI components for consistency
- Implement proper error boundaries

## Development Patterns

### Backend Patterns

#### 1. Actions Pattern
Use actions for complex operations:
```php
namespace App\Actions\Applications;

class RegisterApplicationAction
{
    public function execute(array $data): Application
    {
        // Complex registration logic
    }
}
```

#### 2. Broadcasting Pattern
Create events for real-time updates:
```php
namespace App\Events\Applications;

class ApplicationStatusChanged implements ShouldBroadcast
{
    use Dispatchable, InteractsWithSockets, SerializesModels;
    
    public function broadcastOn(): array
    {
        return [
            new Channel('applications'),
            new PrivateChannel("application.{$this->application->id}"),
        ];
    }
}
```

#### 3. Pulse Integration
Create custom Pulse recorders:
```php
namespace App\Pulse\Recorders;

class ApplicationMetricsRecorder
{
    // Record custom metrics for managed applications
}
```

### Frontend Patterns

#### 1. Composables Pattern
```typescript
// composables/useApplicationStatus.ts
export function useApplicationStatus(applicationId: number) {
  const status = ref<ApplicationStatus>('unknown')
  
  // Listen for Reverb events
  // Update status reactively
  
  return { status }
}
```

#### 2. Component Structure
```vue
<script setup lang="ts">
interface Props {
  application: Application
}

const props = defineProps<Props>()
const emit = defineEmits<{
  'update:status': [status: ApplicationStatus]
}>()
</script>
```

#### 3. Type Definitions
Define shared types in `resources/js/types/`:
```typescript
export interface Application {
  id: number
  name: string
  url: string
  status: ApplicationStatus
  metrics?: ApplicationMetrics
}
```

## Best Practices

1. **Always type your code** - Use TypeScript interfaces and PHP type hints
2. **Broadcast significant events** - Keep clients updated in real-time
3. **Handle WebSocket disconnections** - Implement reconnection logic
4. **Optimize queries** - Use eager loading, avoid N+1 problems
5. **Cache aggressively** - Use Laravel cache for heavy computations
6. **Monitor performance** - Use Pulse to track your own metrics
7. **Write tests first** - TDD for critical features
8. **Document complex logic** - Add comments for non-obvious code

## Troubleshooting

- **Reverb not connecting**: Check REVERB_* environment variables, verify firewall rules
- **Pulse not recording**: Ensure PULSE_ENABLED=true, check recorder configuration
- **Type errors**: Run `npm run types:check` to identify issues
- **SSR errors**: Check console output from `npm run build:ssr`
